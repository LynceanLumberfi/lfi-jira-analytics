# LFI Skill Fingerprint

## Purpose

When a developer pastes a file into a JIRA ticket description, use this guide to determine whether it was generated by a Claude Code skill and, if so, which one. Find the definitive marker first — one exact match is conclusive. If no definitive marker is present, count the strong markers that match; 3 or more from a single skill's list indicates high confidence.

## Two Skills to Detect

| Skill | What it produces | File pasted into JIRA |
| --- | --- | --- |
| `/lfi-prepare-dev-plan` | Structured frontend implementation plan | `docs/implementation/fe-dev-implementation-plan.md` |
| `/lfi-execute-dev-plan` | QA handoff document summarising what was built | `docs/implementation/fe-dev-implementation-summary.md` |

---

## Skill 1 — `/lfi-prepare-dev-plan`

### Definitive marker

The HTML comment on line 3 of the file:

```
<!-- Source: jira-requirement.txt | Generated by /lfi-prepare-dev-plan -->
```

### Strong markers — 3 or more = high confidence

| # | Pattern | Where to look |
| --- | --- | --- |
| 1 | Title is exactly `# Implementation Plan: <Feature Name>` | First line |
| 2 | `## Acceptance Criteria` table whose columns are `#` and `Criterion` | Near top |
| 3 | `## Affected Areas` table whose columns are `Path` and `Role` | After Goal |
| 4 | `## Architecture Notes` table with a `yes / no / partial` value column and rows named e.g. "List virtualization needed (50+ rows)" and "Memoization hotspots" | After Affected Areas |
| 5 | Phase headings follow the pattern `## Phase N — <Name>` (e.g. `## Phase 1 — Types & Constants`) | Body |
| 6 | Every phase contains all seven sub-sections in this fixed order: `### Files` › `### Implementation Details` › `### Types & DTOs` › `### API Changes` › `### State / Store Changes` › `### Components & Hooks` › `### Tests` | Within each phase |
| 7 | Empty sub-sections are rendered as the literal token `_None_` — never omitted | Within phases |
| 8 | `### Files` table has a `Change type` column whose values are `NEW` or `MODIFIED` | Within each phase |
| 9 | `### API Changes` table columns include `baseURL`, `Auth`, and `Cancel token` | API phases |
| 10 | File ends with a `## Conventions Reminder` table whose columns are `#`, `Rule`, and `Source` | Last section |

## Skill 2 — `/lfi-execute-dev-plan`

### Definitive marker

The HTML comment on line 3 of the file:

```
<!-- Source: fe-dev-implementation-plan.md | Generated by /lfi-execute-dev-plan -->
```

### Strong markers — 3 or more = high confidence

| # | Pattern | Where to look |
| --- | --- | --- |
| 1 | Title is exactly `# QA Handoff: <Feature Name>` | First line |
| 2 | `## Acceptance Criteria Checklist` table with columns `#`, `Criterion`, `How to verify`, `Status` | Near top |
| 3 | Status column in the Acceptance Criteria Checklist contains `☐` checkbox symbols | Checklist table |
| 4 | `## Test Scenarios` section with both `### Primary flows` and `### Edge cases` sub-sections | Body |
| 5 | `### Mutation flows` table contains a `Double-submit protected?` column with `yes / no` values | Under Test Scenarios |
| 6 | `## Permissions & Feature Flags` section with columns `Gate`, `Type`, `When visible` — or the placeholder `_No permission or feature flag gating in this feature._` | Body |
| 7 | `## What Was Built` section with both `### New files` and `### Modified files` sub-sections | Body |
| 8 | `## Out of Scope` section — or the placeholder `_Nothing deferred._` | Body |
| 9 | `## How to Run Locally` section contains a `npm start` bash code block | Near end |
| 10 | `## Test Commands` section contains both `npm test -- --testPathPattern=` and `npm run lintfix` | Last section |

## How Claude Should Answer

| Conclusion | When to use it |
| --- | --- |
| `/lfi-prepare-dev-plan` used | Definitive marker 1 present, **or** 3+ strong markers from Skill 1 match |
| `/lfi-execute-dev-plan` used | Definitive marker 2 present, **or** 3+ strong markers from Skill 2 match |
| Both used | Both definitive markers present (description contains both files pasted together) |
| No skill detected | Neither definitive marker present and fewer than 3 strong markers match for either skill |
