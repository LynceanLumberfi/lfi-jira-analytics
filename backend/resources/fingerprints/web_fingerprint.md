# Fingerprint — `/lfi-prepare-dev-plan`

Use this checklist to determine whether a given implementation plan (or related artifact) was produced by the `/lfi-prepare-dev-plan` skill. A document is "very likely generated by this skill" if it matches **all hard markers** and **most soft markers**.

> Note: this skill **reads** `docs/implementation/jira-requirement.txt` (a JIRA ticket) and **writes** `docs/implementation/fe-dev-implementation-plan.md`. The fingerprint below targets the **output plan file** — JIRA tickets themselves are user-authored input and are not produced by this skill.

---

## 1. Hard markers (deterministic — must all match)

| #   | Marker                                                                                                                                                                              | Where it appears                |
| --- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| 1   | File path is exactly `docs/implementation/fe-dev-implementation-plan.md`                                                                                                            | File location                   |
| 2   | First line is `# Implementation Plan: <Feature Name>`                                                                                                                               | H1 heading                      |
| 3   | Contains the HTML comment `<!-- Source: jira-requirement.txt \| Generated by /lfi-prepare-dev-plan -->`                                                                             | Immediately under H1            |
| 4   | Top-level sections in this exact order: `Goal`, `Acceptance Criteria`, `Affected Areas`, `Architecture Notes`, one or more `Phase N — <Name>`, `Conventions Reminder`               | H2 headings                     |
| 5   | Every `Phase N` contains exactly these seven H3 sub-sections in this order: `Files`, `Implementation Details`, `Types & DTOs`, `API Changes`, `State / Store Changes`, `Components & Hooks`, `Tests` | Per-phase schema                |
| 6   | Empty sections render as the literal token `_None_` (italics-underscored), never omitted, never replaced with "N/A" or empty                                                        | Anywhere a section has no items |
| 7   | All itemized content is rendered as **markdown tables** (not bullets, not prose). Only allowed non-table content: the prose `Goal`, fenced TS/JSON code blocks, and `_None_`        | Throughout                      |

If any of markers 2, 3, 4, 5 is missing or reordered, the file was almost certainly **not** generated by this skill.

---

## 2. Soft markers (strong signals — most should match)

| #   | Marker                                                                                                                              |
| --- | ----------------------------------------------------------------------------------------------------------------------------------- |
| 1   | `Acceptance Criteria` table has columns `#` and `Criterion`                                                                         |
| 2   | `Affected Areas` table has columns `Path` and `Role`                                                                                |
| 3   | `Architecture Notes` table has columns `Concern`, `Applies`, `Notes` and includes rows like *New route / Launcher entry*, *Global state (Easy Peasy model)*, *Backend endpoint exists*, *Permission / feature flag gating*, *List virtualization needed (50+ rows)*, *Memoization hotspots*, *Sentry capture on API errors*, *i18n / copy in constants file* |
| 4   | `Files` table columns: `Path`, `Change type`, `What changes`; `Change type` values are exactly `NEW` or `MODIFIED`                  |
| 5   | `Types & DTOs` table columns: `Name`, `Kind`, `Fields / Members`, `Notes`; `Kind` values are exactly `interface`, `type`, or `enum` |
| 6   | `API Changes` table columns: `Function`, `Method`, `URL`, `baseURL`, `Auth`, `Request`, `Response`, `Cancel token`                  |
| 7   | API function names are prefixed with `api` (e.g. `apiGetProjects`)                                                                  |
| 8   | `baseURL` values are exactly `platformApiUrl` or `payrollApiUrl`                                                                    |
| 9   | `Auth` column references `getPlatformHeaders()`                                                                                     |
| 10  | `State / Store Changes` table columns: `Model file`, `Change type`, `Actions`, `Thunks`, `Selectors`, `Subscribers`                 |
| 11  | `Components & Hooks` table columns: `Name`, `Kind`, `Path`, `Props / Signature`, `Responsibility`, `Memoization`; `Kind` is one of `Screen`, `Container`, `Presentational`, `Layout`, `Hook` |
| 12  | `Tests` table columns: `Type`, `File`, `Key scenarios`; `Type` values are exactly `Unit` or `Component`                             |
| 13  | Test file names colocated under `__tests__/` and use `*.test.ts` or `*.test.tsx`                                                    |
| 14  | `Conventions Reminder` table columns: `#`, `Rule`, `Source`; `Source` cites `docs/ai-context/<file>.md` or `docs/claude-skills/context/coding_guidelines.md` |
| 15  | Phases are typically titled in this canonical order: *Types & Constants* → *API Module* → *State Model* → *Hooks & Components* → *Screen Wiring* → *Tests* (some may be skipped) |
| 16  | Directory references favor LumberFi-web layout: `src/screens/<Feature>/`, `src/api/<domain>/`, `src/models/<domain>Model.ts`, `src/types/<domain>/`, `src/components/`, `src/hooks/`, `src/utils/`, `src/constants/` |

## 3. Anti-markers (presence suggests *not* this skill)

| #   | Marker                                                                                                  |
| --- | ------------------------------------------------------------------------------------------------------- |
| 1   | Bullet lists (`- foo`) used for itemized phase content instead of tables                                |
| 2   | `N/A`, `TBD`, `—` (em dash) or empty body where this skill would emit `_None_`                          |
| 3   | Missing any of the seven mandatory phase sub-sections                                                   |
| 4   | Different phase sub-section ordering (e.g. `Tests` before `Files`)                                      |
| 5   | API functions without the `api` prefix, or HTTP calls referencing a `baseURL` other than `platformApiUrl` / `payrollApiUrl` |
| 6   | Mobile/native conventions (e.g. `react-native`, `screens/<Feature>.tsx` flat files) — this skill targets the web SPA only |
| 7   | Plan written to a path other than `docs/implementation/fe-dev-implementation-plan.md`                   |
| 8   | Frontmatter present at top of plan file (this skill emits no frontmatter on the output)                 |

---

## 4. Quick verification procedure

1. Open the candidate file.
2. Check Hard markers 2 and 3 (H1 line + HTML source comment) — these alone are very strong evidence.
3. Verify the seven mandatory phase sub-sections exist in order under at least one `Phase N` heading.
4. Spot-check that empty sections use `_None_` and that itemized content is tabular.
5. Confirm column headers in 2-3 random tables match the soft markers above.

If steps 2–4 all match, the artifact was almost certainly generated by `/lfi-prepare-dev-plan`.
