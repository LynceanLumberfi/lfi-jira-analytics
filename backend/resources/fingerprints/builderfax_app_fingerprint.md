# BuilderFax Skill Fingerprint

## Purpose

This file lets Claude identify — from a JIRA ticket description alone — whether a BuilderFax Claude Code skill was used to generate the content and which skill produced it. Developers run skills in this repo, then copy the output file into a JIRA ticket description. When later asked to work from a JIRA ticket, Claude should scan the description for the markers below before doing any analysis: a single definitive marker is conclusive; three or more strong markers together constitute high confidence. Once a skill is identified, Claude should treat the pasted content as structured skill output rather than freeform requirements text.

## Two Skills to Detect

| Skill | What it produces | File pasted into JIRA |
|-------|------------------|-----------------------|
| `/lfi-prepare-dev-plan` | Phase-by-phase implementation plan | `docs/implementation/app-dev-implementation-plan.md` |
| `/lfi-execute-dev-plan` | QA handoff document | `docs/implementation/app-dev-implementation-summary.md` |

---

## Skill 1 — `/lfi-prepare-dev-plan`

### Definitive marker

An HTML comment on the second line of the document:

```
<!-- Source: jira-requirement.txt | Generated by /lfi-prepare-dev-plan -->
```

If this comment appears anywhere in the JIRA description, the content was produced by `/lfi-prepare-dev-plan`. No other skill or human author writes this exact string.

### Strong markers — 3 or more = high confidence

| # | Pattern | Where to look |
|---|---------|---------------|
| 1 | H1 title in the form `# Implementation Plan: <Feature Name>` | First line of the document |
| 2 | `## Acceptance Criteria` section containing a two-column table with headers `#` and `Criterion` (no Status or checkbox column) | After `## Goal` |
| 3 | `## Affected Areas` table with columns `Path` and `Role` listing paths like `` `app/<feature>/` ``, `` `components/screens/<Feature>/` ``, `` `scripts/api/<domain>/` `` | After `## Acceptance Criteria` |
| 4 | `## Architecture Notes` table with three columns `Concern`, `Applies`, `Notes` and rows for "Platform targets", "New route in expo-router", "WebView / iframe embedding compatibility needed" | After `## Affected Areas` |
| 5 | Phase headings in the exact format `## Phase N — <Name>` (e.g. `## Phase 1 — Types & Constants`, `## Phase 2 — API module(s)`) | Body of the document |
| 6 | Every phase contains a `### Files` sub-section with a three-column table headed `Path`, `Change type`, `What changes` | Inside each phase section |
| 7 | Every phase contains a `### State / Store Changes` sub-section whose table includes a `Persist allowlist` column with `yes / no` values | Inside each phase section |
| 8 | Every phase contains a `### Components & Hooks` sub-section whose table includes a `Memoization / Platform notes` column | Inside each phase section |
| 9 | Empty sub-sections rendered as the italic placeholder `_None_` rather than being omitted | Inside any phase sub-section |
| 10 | `## Conventions Reminder` table at the end of the document with columns `#`, `Rule`, `Source` — source values reference `` `CLAUDE.md` `` or `` `docs/claude-skills/context/coding_guidelines.md` `` | Last section of the document |

## Skill 2 — `/lfi-execute-dev-plan`

### Definitive marker

An HTML comment on the second line of the document:

```
<!-- Source: app-dev-implementation-plan.md | Generated by /lfi-execute-dev-plan -->
```

If this comment appears anywhere in the JIRA description, the content was produced by `/lfi-execute-dev-plan`. No other skill or human author writes this exact string.

### Strong markers — 3 or more = high confidence

| # | Pattern | Where to look |
|---|---------|---------------|
| 1 | H1 title in the form `# QA Handoff: <Feature Name>` | First line of the document |
| 2 | `## Acceptance Criteria Checklist` section with a four-column table headed `#`, `Criterion`, `How to verify`, `Status` — where every Status cell contains the `☐` checkbox character | After `## Feature` |
| 3 | `## Test Scenarios` section containing exactly three sub-sections: `### Primary flows`, `### Edge cases`, `### Mutation flows (if any)` | Middle of the document |
| 4 | `### Edge cases` table with pre-populated scenario rows: "empty state", "loading state", "network error", "permission denied" | Inside `## Test Scenarios` |
| 5 | `### Mutation flows (if any)` table with a `Double-submit protected?` column containing `yes / no` values | Inside `## Test Scenarios` |
| 6 | `## Permissions & Feature Flags` section that uses the blockquote placeholder `_No permission or feature flag gating in this feature._` when empty | After `## Test Scenarios` |
| 7 | `## What Was Built` section split into `### New files` and `### Modified files` sub-sections | After `## Permissions & Feature Flags` |
| 8 | `### New files` table with a `Kind` column whose values are drawn from: `Route / Screen / Component / Hook / API / Model / Constant / Test` | Inside `## What Was Built` |
| 9 | `## Out of Scope` section that uses the blockquote placeholder `_Nothing deferred._` when empty | After `## What Was Built` |
| 10 | `## Platform-Specific Notes` table with fixed rows for `iOS`, `Android`, and `Web` — or the placeholder `_No platform-specific behavior in this feature._` | Last section of the document |
