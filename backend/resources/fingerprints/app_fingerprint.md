# LumberFI Implementation Workflow Skill Fingerprint

## Purpose

This file lets Claude identify — from a JIRA ticket description alone — whether a LumberFI skill was used to produce its content and which one. Each skill writes output with a predictable structure: a unique HTML comment (definitive) or a combination of characteristic section names, table column patterns, and placeholder conventions (strong). A single definitive marker is conclusive. Three or more strong markers together indicate high confidence. Use this file when triaging a JIRA ticket to understand what work has been done, which tool generated the plan or report attached, and how to interpret the content.

---

## Four Skills to Detect

| Skill name | What it produces | Content pasted into JIRA from |
| --- | --- | --- |
| `lfi-prepare-dev-plan` | `docs/implementation/app-dev-implementation-plan.md` | Implementation plan (phases, types, API, state, components) |
| `lfi-execute-dev-plan` | `docs/implementation/app-dev-implementation-summary.md` | QA handoff doc (acceptance checklist, test scenarios, what was built) |
| `lfi-code-review` | Terminal output only (no file) | Code review table copied from terminal |
| `qa-validate` | Terminal output only (no file) | QA validation report copied from terminal |

## Skill 3 — lfi-code-review

### Definitive marker

None — this skill prints directly to the terminal and writes no file. There is no HTML comment or unique heading. Use 3 or more strong markers to identify its output.

### Strong markers — 3 or more = high confidence

| # | Pattern | What to match |
| --- | --- | --- |
| 1 | No preceding header | Content begins directly with a markdown table — no title line, no feature name, no HTML comment above the table |
| 2 | Four-column table | Exactly these columns in order: `File`, `Issue`, `Suggested Fix`, `Severity` — no other columns |
| 3 | `Severity` vocabulary | Values restricted to exactly four words: `Critical`, `High`, `Medium`, `Low` — no other values appear |
| 4 | Summary line separator | Trailing one-line count uses middle dot `·` as separator: e.g. `3 Critical · 2 High · 4 Medium · 1 Low` |
| 5 | "No issues found" phrasing | Variant output reads: `No issues found — all reviewed changes comply with the guidelines in \`docs/claude-skills/context/coding_guidelines.md\`.` |
| 6 | `File` column content | Values are source file paths relative to the repo root (e.g. `src/screens/...`, `src/models/...`) — not JIRA fields or test IDs |
| 7 | Flat structure | Report has no section headings (`##`, `###`) — just the table and the summary line, nothing else |
| 8 | Severity definition wording | If the review is included with annotation, severity descriptions use the phrases `hard architectural rule`, `correctness / security risk`, and `runtime issues, data loss` |

## Skill 4 — qa-validate

### Definitive marker

`**Input mode:**` bold label in the header block. This label is written by no other skill and takes one of exactly four fixed values: `auto-detected from current branch`, `branch provided by user`, `ticket provided directly by user`, or `prompted user for input`. Its presence alone confirms `qa-validate` output.

### Strong markers — 3 or more = high confidence

| # | Pattern | What to match |
| --- | --- | --- |
| 1 | Title | `# QA Validation Report` with no feature name after the title (contrast with `# QA Handoff: <name>` from `lfi-execute-dev-plan`) |
| 2 | Header block labels | Eight consecutive `**Label:**` lines: `Branch / Ref`, `Parent Ticket`, `FE Ticket`, `BE Ticket`, `QA Ticket`, `Commits`, `Files changed`, `Base branch` |
| 3 | `**Parent Ticket:** TIM-` prefix | Ticket key always starts with `TIM-` followed by digits; this pattern appears in the header block |
| 4 | `**FE Ticket:**` and `**BE Ticket:**` labels | These two labels — classifying child tickets — are unique to this skill's header |
| 5 | `## Summary` table | Columns in order: `Total`, `Passed`, `Failed`, `At Risk` followed immediately by a `**Pass rate:** N%` line |
| 6 | `## Test Case Results` table | `Status` column whose values are restricted to three all-caps words: `PASS`, `FAIL`, `RISK` |
| 7 | `## Regression Risks` section | Risk items include parenthesised severity ratings using all-caps: `(LOW)`, `(MEDIUM)`, or `(HIGH)` |
| 8 | `## Failed & At-Risk Test Details` section | Each entry contains a `**Suggestion:**` bold sub-label — this label appears in no other skill output |
| 9 | `## Files Changed (grouped)` section | Area names presented as bold labels: `**Screens:**`, `**Models:**`, `**Hooks:**`, `**Utils:**`, `**API:**`, `**Other:**` |
| 10 | No file path — terminal-only | Report does not reference `docs/implementation/` as an output path; its own content says it is printed to terminal and not saved |
