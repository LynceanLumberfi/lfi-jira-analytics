# Backend Skill Fingerprint

## Purpose
Use this file to identify whether a backend Claude Code skill was used to write the content found in a JIRA ticket description.
Send this file plus the JIRA description to Claude and ask:
**"Based on this fingerprint, was a backend skill used? If yes, which one — Prepare Dev Plan or Execute Dev Plan?"**

---

## Two Skills to Detect

| Skill | What it produces | File pasted into JIRA |
|-------|-----------------|----------------------|
| **Prepare Dev Plan** | A structured implementation plan | `be-dev-plan-implementation.md` |
| **Execute Dev Plan** | A UI-consumable API/DTO summary | `be-dev-implementation-summary.md` |

---

## Skill 1 — Prepare Dev Plan

Developers paste the **implementation plan** into the JIRA description after running this skill.

### Definitive marker (one match = confirmed)
The description contains this exact HTML comment:
```
<!-- Source: jira-requirement.txt | Generated by /lfi-prepare-dev-plan -->
```

### Strong markers — 3 or more = high confidence the skill was used

| # | What to look for |
|---|-----------------|
| 1 | Document title is `# Implementation Plan:` followed by a feature name |
| 2 | Contains `## Acceptance Criteria` with a numbered markdown table |
| 3 | Contains `## Affected Modules` with a table of `Module \| Role` columns |
| 4 | Contains `## Architecture Notes` with a table of `Concern \| Applies \| Notes` columns |
| 5 | Each phase heading follows `## Phase N — <Name>` and contains exactly these 7 sub-sections in order: `### Files`, `### Implementation Details`, `### DB Changes`, `### API Changes`, `### DTO Changes`, `### Events`, `### Tests` |
| 6 | Empty sub-sections contain the literal text `_None_` — never blank, never omitted |
| 7 | All itemized content is in markdown tables — no bullet lists, no prose |
| 8 | Document ends with a `## Conventions Reminder` section |
| 9 | Migration filename contains `XXXXXX-description.sql` as a placeholder |
| 10 | References backend module names such as `lumberfi-db`, `lumberfi-api`, `lumberfi-web`, `lumberfi-public-api` |

## Skill 2 — Execute Dev Plan

Developers paste the **implementation summary** into the JIRA description after running this skill.

### Definitive marker (one match = confirmed)
The description contains this exact HTML comment:
```
<!-- Source: be-dev-plan-implemetation.md | Generated by /lfi-execute-dev-plan | Consumed by UI team -->
```

### Strong markers — 3 or more = high confidence the skill was used

| # | What to look for |
|---|-----------------|
| 1 | Document title is `# Implementation Summary:` followed by a feature name |
| 2 | Contains all of these top-level sections: `## Feature`, `## API Changes`, `## New & Modified DTOs`, `## Breaking Changes`, `## Events`, `## Files Touched`, `## Suggested UI Next Steps` |
| 3 | API endpoints are introduced as ` ### \`METHOD /api/v1/...\` — short description` |
| 4 | Each endpoint has an attribute table with rows for `Method`, `URL`, `Auth`, `Status` |
| 5 | DTO field tables have these exact columns: `Field (snake_case in JSON) \| Java type \| Nullable \| Enum values (if enum) \| Description` |
| 6 | Empty sections use italic placeholder sentences such as `_No API changes in this feature._`, `_No breaking changes._`, `_No DTO changes in this feature._`, `_No events exposed to UI in this feature._` |
| 7 | `## Files Touched` section has `**Created:**` and `**Modified:**` sub-headers |
| 8 | `## Suggested UI Next Steps` is present and its first item mentions regenerating TypeScript types |
| 9 | Each endpoint section contains `**Example request:**` and `**Example response:**` fenced JSON blocks |
| 10 | DTO headings include a status tag: ` ### \`DTOName\` — \`NEW\`` or ` ### \`DTOName\` — \`MODIFIED\`` |

## How Claude Should Answer

After reading the JIRA description against the markers above:

1. **Check for definitive markers first.** If either HTML comment is present → confirmed, state the skill name.
2. **Count strong markers** for each skill. 3 or more → high confidence. 1–2 → low confidence.
3. **Report the result** using one of these conclusions:

| Conclusion | When |
|-----------|------|
| **Prepare Dev Plan skill used** | Definitive marker found, OR 3+ strong markers from Skill 1 |
| **Execute Dev Plan skill used** | Definitive marker found, OR 3+ strong markers from Skill 2 |
| **Both skills used** | Both sets of markers are present (developer pasted both documents) |
| **No backend skill detected** | Fewer than 3 strong markers from either skill; content appears manually written |
